1. What is MergeBetter?

MergeBetter is a Jira Cloud app that prevents GitHub pull requests from merging until the linked Jira issue meets conditions you define.

Think of it as a quality gate between your Jira workflow and your GitHub branches.

Example use cases:

  • Block merges to main unless the Jira ticket is marked "Done"
  • Prevent releases to release-* unless the fix version matches the release
  • Stop hotfix merges unless the issue type is "Bug" and status is "In Review"
  • Ensure no story is merged unless it has been assigned to the correct sprint

2. How It Works

Developer opens Pull Request on GitHub | v GitHub Actions workflow runs automatically | v MergeBetter webhook checks the linked Jira issue (issue key extracted from the PR title, e.g. "NEON-123") | +--- Issue meets policy conditions? ---> [PASS] PR check passes -> merge allowed | \--- Issue does NOT meet conditions? --> [FAIL] PR check fails -> merge blocked (with your custom rejection message)

Key points:

  • The Jira issue key must be included in the PR title (e.g. NEON-123 Fix login bug)
  • MergeBetter checks the issue against your JQL rules
  • The result appears as a GitHub status check on the pull request
  • No code or credentials are stored outside Atlassian's infrastructure

3. Installation

Step 1 - Install from Atlassian Marketplace

  1. Go to your Jira Cloud instance (e.g. yourcompany.atlassian.net)
  2. Click Apps in the top navigation -> Explore more apps
  3. Search for "MergeBetter"
  4. Click Get app -> Get it now
  5. Confirm the installation
Who can install? Only Jira administrators can install Marketplace apps.

Step 2 - Verify Installation

After installation:

  1. Go to any Jira project
  2. Click Project settings (bottom-left sidebar)
  3. Scroll down to the Apps section
  4. You should see "Merge Policies" listed

4. Opening the App

  1. Navigate to your Jira project
  2. Click Project settings in the left sidebar
  3. Under Apps, click Merge Policies

You will see the Merge Policies dashboard listing all policies for this project.

5. Creating Your First Policy

Step 1 - Click "+ Add Policy"

On the Merge Policies dashboard, click the + Add Policy button (top-left).

Step 2 - Fill in Policy Details

FieldRequiredDescription
NameRequiredA short label for this policy (e.g. Main Branch Policy)
DescriptionOptionalNotes about what this policy does and which repos use it

Step 3 - Add Rules

Each policy can have one or more rules. Click + Add Rule to add your first rule. Each rule has three parts:

ColumnDescriptionExample
Target Branch PatternThe GitHub branch this rule applies tomain, release-*, hotfix-*
Rejection MessageWhat the developer sees if the check failsTicket must be In Progress with fix version 2.1.0
JQL QueryThe condition the Jira issue must satisfystatus = "In Progress" AND fixVersion = "2.1.0"

Step 4 - Test Your JQL

Before saving, click Test JQL next to each rule to validate your query.

  • Green - JQL is valid and will work correctly
  • Red - JQL has an error; the error message tells you exactly what to fix

Step 5 - Save

Click Save Changes (top-right or bottom of the form). Your policy is now active and ready to connect to GitHub.

6. Understanding Rules

Branch Pattern Matching

The Target Branch Pattern supports wildcards using *.

PatternMatches
mainOnly the main branch
release-*release-2.1.0, release-3.0, release-hotfix
hotfix-*hotfix-login, hotfix-payment-bug
feature-*feature-dark-mode, feature-NEON-99
Tip: Use release-* to cover all release branches with one rule instead of creating a separate rule for each release.

JQL Query

The JQL query is evaluated against the Jira issue linked in the PR title. The issue must match the JQL for the PR check to pass.

For example, if your JQL is:

status = "In Progress" AND fixVersion = "2.1.0"

Then the issue must be In Progress AND have fix version 2.1.0 for the PR to be allowed to merge.

Rejection Message

This is what developers see in GitHub when their PR is blocked. Make it clear and actionable:

  • Bad   Policy violation
  • Good   The Jira ticket must be "In Progress" with fix version 2.1.0 before merging to release-2.1.0

Multiple Rules in One Policy

You can add multiple rules to a single policy. All matching rules must pass for the PR to be allowed. A rule only applies if its Target Branch Pattern matches the PR's target branch.

#Target BranchJQLPurpose
1mainstatus = "Done"Main branch: ticket must be Done
2release-*fixVersion is not EMPTY AND status = "In Progress"Release branches: must have a fix version
3hotfix-*issuetype = Bug AND priority in (High, Critical)Hotfixes: only high-priority bugs

7. Managing Policies

Enable / Disable a Policy

On the dashboard, each policy has a toggle switch:

  • ACTIVE - Policy is enforced on all PRs
  • DISABLED - Policy is paused; PRs are not checked

Use this to temporarily suspend a policy without deleting it (e.g. during a hotfix emergency).

Edit a Policy

Click Edit next to any policy to open the editor and modify its name, description, or rules.

Delete a Policy

Click Delete next to a policy to permanently remove it.

[!] Warning: Deleting a policy cannot be undone. Any GitHub Actions workflows pointing to this policy's ID will no longer have a matching policy and will pass by default.

8. Webhook URL - Connecting to GitHub

Each Jira installation gets a unique webhook URL - this is the endpoint GitHub Actions calls to trigger a policy check.

Finding Your Webhook URL

  1. Go to Merge Policies in your project settings
  2. Click Show Webhook URL (top-right of the dashboard)
  3. The URL will appear with a Copy button
  4. A ready-to-use GitHub Actions workflow snippet is also shown
Your URL looks like:
https://webhook.forge.atlassian.com/webtrigger/...

Security

  • The webhook URL acts as a shared secret - only people with the URL can trigger checks
  • Store it as a GitHub Actions secret - never hardcode it in your workflow file
  • Always set policyId in every repo's workflow - this is required to isolate repos from each other

9. GitHub Actions Integration

Step 1 - Add Webhook URL as a GitHub Secret

  1. Go to your GitHub repository
  2. Click Settings -> Secrets and variables -> Actions
  3. Click New repository secret
  4. Name: MERGE_BETTER_WEBHOOK_URL
  5. Value: (paste the webhook URL from the app)
  6. Click Add secret

Step 2 - Find Your Policy ID

Every workflow must include a policyId to specify which policy to enforce.

  1. Go to Merge Policies in your Jira project settings
  2. The ID column on the dashboard shows the policy number (e.g. 1, 2, 3)
  3. Note the ID of the policy you want this repo to enforce
[!] policyId is required. Workflows submitted without a policyId will be rejected.

Step 3 - Add the Workflow File

Create a file at .github/workflows/merge-check.yml in your repository.
Replace YOUR_POLICY_ID with the Policy ID from Step 2:

# .github/workflows/merge-check.yml
name: MergeBetter Jira Policy Check

on:
  pull_request:
    types: [opened, synchronize, reopened, edited]

permissions:
  pull-requests: write   # needed to post a comment on the PR

jobs:
  jira-policy-check:
    runs-on: ubuntu-latest
    steps:
      - name: Extract Jira Issue Key from PR Title
        id: extract
        env:
          PR_TITLE:  "${{ github.event.pull_request.title }}"
          GH_TOKEN:  "${{ github.token }}"
          PR_NUMBER: "${{ github.event.pull_request.number }}"
        run: |
          ISSUE_KEY=$(echo "$PR_TITLE" \
            | grep -oE '[A-Z]+-[0-9]+' | head -1)
          if [ -z "$ISSUE_KEY" ]; then
            echo "[X] No Jira issue key found in PR title."
            {
              echo "## MergeBetter: No Jira Issue Key Found"
              echo ""
              echo "Please add a Jira issue key to your PR title."
              echo ""
              echo "| Field | Value |"
              echo "|-------|-------|"
              echo "| Current title | \`$PR_TITLE\` |"
              echo "| Expected format | \`PROJ-123 Your PR description\` |"
            } | gh pr comment "$PR_NUMBER" --repo "$GITHUB_REPOSITORY" --body-file -
            exit 1
          fi
          echo "[OK] Found Jira issue key: $ISSUE_KEY"
          echo "issue_key=$ISSUE_KEY" >> $GITHUB_OUTPUT

      - name: Check Jira Merge Policy
        env:
          ISSUE_KEY:  "${{ steps.extract.outputs.issue_key }}"
          BASE_REF:   "${{ github.base_ref }}"
          WEBHOOK_URL: "${{ secrets.MERGE_BETTER_WEBHOOK_URL }}"
          GH_TOKEN:   "${{ github.token }}"
          PR_NUMBER:  "${{ github.event.pull_request.number }}"
        run: |
          JSON_BODY="{\"issueKey\":\"$ISSUE_KEY\",\"branch\":\"$BASE_REF\",\"policyId\":YOUR_POLICY_ID}"
          RESPONSE=$(curl -s -w "\n%{http_code}" \
            -X POST \
            -H "Content-Type: application/json" \
            -d "$JSON_BODY" \
            "$WEBHOOK_URL")

          HTTP_CODE=$(echo "$RESPONSE" | tail -1)
          BODY=$(echo "$RESPONSE" | sed '$d')

          echo "-- MergeBetter Response --"
          echo "$BODY" | jq . 2>/dev/null || echo "$BODY"
          echo "--------------------------"

          ALLOWED=$(echo "$BODY" | jq -r '.allowed // false')

          if [ "$ALLOWED" = "true" ]; then
            echo "[OK] All merge policies passed. PR is clear to merge."
            {
              echo "## MergeBetter: All checks passed"
              echo "Issue **$ISSUE_KEY** satisfies all merge policies for branch **$BASE_REF**."
            } >> $GITHUB_STEP_SUMMARY
          else
            echo "[X] Merge blocked by MergeBetter policy:"
            echo "$BODY" | jq -r '.failures[] | "  Policy : \(.policy)\n  Branch : \(.rule)\n  Reason : \(.reason)"'

            # Post a comment directly on the PR conversation tab
            {
              echo "## MergeBetter: Merge Blocked"
              echo ""
              echo "Issue **$ISSUE_KEY** does not meet the merge policy for branch **$BASE_REF**."
              echo ""
              echo "| Policy | Branch Rule | Reason |"
              echo "|--------|-------------|--------|"
              echo "$BODY" | jq -r '.failures[] | "| \(.policy) | \(.rule) | \(.reason) |"'
            } | gh pr comment "$PR_NUMBER" --repo "$GITHUB_REPOSITORY" --body-file -

            # Also write to the step summary (visible via the "Details" link)
            {
              echo "## MergeBetter: Merge Blocked"
              echo ""
              echo "| Policy | Branch Rule | Reason |"
              echo "|--------|-------------|--------|"
              echo "$BODY" | jq -r '.failures[] | "| \(.policy) | \(.rule) | \(.reason) |"'
            } >> $GITHUB_STEP_SUMMARY

            exit 1
          fi
Example with Policy ID 1: Replace YOUR_POLICY_ID with 1 in the workflow above. Do not add quotes around the number - it must be a JSON integer.

Step 4 - Run the Workflow Once First

[!] Required before Step 5: GitHub only shows a status check in the Branch Protection search box after it has run at least once on the repository. You must trigger the workflow before you can add it as a required check.

To trigger the first run:

  1. Make sure the .github/workflows/merge-check.yml file is committed and pushed to your default branch
  2. Open any pull request in the repository (the PR title must include a Jira issue key, e.g. NEON-123 Test run)
  3. Wait for the jira-policy-check workflow to run - you will see it appear under the PR's checks
  4. Once it has run (pass or fail), GitHub registers the check name and it becomes searchable in Branch Protection settings

Step 5 - Block the Merge Button (Branch Protection Rules)

The workflow failing alone does not prevent developers from clicking Merge. You must configure GitHub Branch Protection.

  1. Go to your GitHub repository -> Settings -> Branches
  2. Click Add branch protection rule (or edit an existing one)
  3. Set Branch name pattern to main (repeat for release-*, hotfix-*, etc.)
  4. Enable "Require status checks to pass before merging"
  5. In the search box, type jira-policy-check - it will now appear in the list; select it
  6. Enable "Do not allow bypassing the above settings"
  7. Click Save changes
Check not appearing? It means the workflow has not run yet on this repo. Go back to Step 4 and open a test PR first.

The merge button will now be locked until the policy check passes:

Some checks were not successful
  [X] jira-policy-check - Merge blocked: Ticket must be In Progress

  [ Merging is blocked ]   <- button disabled
Repeat Steps 4-5 for each protected branch (main, release-*, develop, etc.).

Step 5 - PR Title Format

Developers must include the Jira issue key in their PR title:

PR TitleIssue Key Detected
NEON-123 Fix login bug[OK] NEON-123
[NEON-456] Add dark mode[OK] NEON-456
Fix login bug[X] No key found - check fails immediately
fix/NEON-789-payment[X] Not detected (branch name is ignored)
Team convention: Ask developers to always start PR titles with the Jira issue key: ABC-123 Short description

10. JQL Query Examples

Basic Status Check

status = "In Progress"

Issue must currently be In Progress.

Status + Fix Version

status = "In Progress" AND fixVersion = "2.1.0"

Issue must be In Progress AND assigned to fix version 2.1.0.

Done or In Review

status in ("In Review", "Done")

Issue must be either In Review or Done.

Specific Issue Types Only

issuetype in (Bug, Story) AND status = "In Progress"

Only Bugs and Stories are allowed, and they must be In Progress.

Sprint-based Validation

sprint in openSprints() AND status != "To Do"

Issue must be in the current open sprint and not sitting in To Do.

Fix Version + Issue Type

fixVersion = "3.0.0" AND issuetype = Story AND status = "Done"

Only Done Stories targeting version 3.0.0.

Priority Gate

priority in (High, Critical) AND status = "In Progress"

Only high or critical priority issues actively being worked on.

Assigned Issues Only

assignee is not EMPTY AND status = "In Progress"

Issue must be assigned to someone and In Progress.

Tip: Test your JQL in Jira's issue search (/issues) first, then paste it into MergeBetter's JQL field and click Test JQL to confirm it is valid.

11. Multiple Policies & Repos

You can create multiple policies in the same Jira project - one per team, repo, or release track.

Example Setup

Policy IDPolicy NameUsed By
1Frontend Policyfrontend-app GitHub repo
2Backend Policybackend-api GitHub repo
3Release PolicyAll repos during release freeze

Targeting Policies Per Repo

Each GitHub repo's workflow must include the policyId of the policy it enforces.

frontend-app workflow (policyId: 1):

JSON_BODY="{\"issueKey\":\"$ISSUE_KEY\",\"branch\":\"$BASE_REF\",\"policyId\":1}"

backend-api workflow (policyId: 2):

JSON_BODY="{\"issueKey\":\"$ISSUE_KEY\",\"branch\":\"$BASE_REF\",\"policyId\":2}"
[!] policyId is required. Always set policyId explicitly in every workflow. This prevents a repo from being accidentally affected by a policy intended for a different team.

12. Frequently Asked Questions

Q: Where do I find the Policy ID?
A: On the Merge Policies dashboard, the first column is ID (1, 2, 3...). Use this number as the policyId in your workflow.

Q: What happens if there is no Jira issue key in the PR title?
A: The check will fail with a message indicating no issue key was found. Instruct your team to always include the key in the PR title (e.g. NEON-123 Fix login page).

Q: Can I have the check pass if there's no matching policy for the branch?
A: Yes. If no active policy has a rule matching the PR's target branch, MergeBetter returns a pass by default. Only matching rules are enforced.

Q: Can I test a policy without blocking real PRs?
A: Yes - create a policy, leave it DISABLED, and enable it only when you're ready to enforce it.

Q: Does MergeBetter store my Jira data?
A: No. Jira issue data is read in real-time per request and never stored. Only your policy configuration (names, branch patterns, JQL rules) is stored - within Atlassian's own Forge infrastructure.

Q: Does MergeBetter work with Jira Server or Data Center?
A: No. MergeBetter is built on Atlassian Forge and works with Jira Cloud only.

Q: Can multiple GitHub repositories use the same webhook URL?
A: Yes. Each repo must set policyId in its workflow to target its specific policy. This is required - not optional.

Q: What if my JQL references a field that doesn't exist in my project?
A: The check will fail with a JQL error. Use the Test JQL button in the policy editor to validate your query before saving.

13. Troubleshooting

[X] PR check fails with "No Jira issue key found in PR title"

Cause: The PR title doesn't contain a valid Jira issue key.
Fix: Rename the PR title to include the issue key. Format: ABC-123 Your description

[X] PR check fails with "Issue does not match policy"

Cause: The Jira issue was found but doesn't satisfy the JQL rule.
Fix: Read the rejection message - it tells you exactly what needs to change on the Jira ticket (e.g. update the status, fix version, or issue type).

[X] PR check fails with "Policy not found"

Cause: The policyId in the workflow doesn't match any active policy.
Fix: Check the policy ID on the Merge Policies dashboard. Update the policyId in your GitHub Actions workflow.

[X] GitHub Actions step fails (workflow error, not a policy failure)

Cause: The webhook URL secret may be missing or incorrect.
Fix:

  1. Go to GitHub repo -> Settings -> Secrets -> Actions
  2. Verify MERGE_BETTER_WEBHOOK_URL is set
  3. Copy the URL again from the app (Show Webhook URL) and update the secret

[X] "Test JQL" shows error: "Expecting a field name at the end of the query"

Cause: Your JQL is syntactically incomplete - often a trailing AND, OR, or missing value.
Fix: Review the JQL for incomplete clauses. Example of bad JQL: status = "In Progress" OR (trailing OR is invalid).

[X] Policy is active but PRs are not being checked

Cause: The GitHub Actions workflow may not be set up, or it's only running on certain events.
Fix: Ensure your workflow triggers include edited:

on:
  pull_request:
    types: [opened, synchronize, reopened, edited]

The edited trigger is important - it re-runs the check when the PR title is updated.

[X] PR check passes but the merge button is still not blocked

Cause: Branch Protection Rules have not been configured in GitHub.
Fix:

  1. Go to GitHub repo -> Settings -> Branches
  2. Add or edit a branch protection rule for main
  3. Enable "Require status checks to pass before merging"
  4. Search for and add jira-policy-check as a required check
  5. Save changes

[X] "jira-policy-check" does not appear in GitHub Branch Protection search

Cause: GitHub only lists a status check as a searchable option after it has run at least once on the repository. If you have never opened a PR with the workflow in place, the check name is unknown to GitHub.
Fix:

  1. Confirm the workflow file (.github/workflows/merge-check.yml) is pushed to your default branch
  2. Open a test pull request with a Jira issue key in the title (e.g. NEON-1 test)
  3. Let the jira-policy-check job run (it can pass or fail - either registers the name)
  4. Return to Settings -> Branches -> Branch protection rule and search again

[X] App not visible in Project Settings

Cause: You may not have Jira admin permissions, or the app isn't installed.
Fix: Ask your Jira administrator to install MergeBetter from the Atlassian Marketplace.
